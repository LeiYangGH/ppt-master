# 主题研究工作流

> **独立预处理工作流** — 产出可输入主PPT生成流水线的源材料，也可独立使用。

## 触发条件

用户**仅提供主题名称、简要描述或一组需求** — 无PDF、DOCX、URL或其他源文件。

示例：
- "做一个关于久石让的PPT"
- "创建一个关于可再生能源趋势的演示文稿"
- "我想介绍我们的新产品（附带简要描述）"

## 交付物

所有输出必须放在 `workspace/sources/` 目录下：

| 交付物 | 路径 | 示例 |
|--------|------|------|
| 结构化Markdown文档 | `workspace/sources/<topic_name>.md` | `workspace/sources/joe_hisaishi.md` |
| 图片文件夹 | `workspace/images/` | `workspace/images/` |

> **命名一致性规则**：Markdown文件名（不含 `.md`）与图片文件夹名必须完全相同。
>
> **输出目录规则**：文档和图片文件夹必须在 `workspace/` 内创建，不得放在仓库根目录或其他位置。

## 流程概览

```
确认主题 → 研究内容 → 收集图片 → 输出
```

## ⛔ 全局硬性约束（适用于所有搜索与图片步骤）

| # | 约束 | 说明 |
|---|------|------|
| 1 | **搜索关键字必须用中文** | 所有 `web_search.py` 调用中的 query、以及 IDE 内置搜索工具的 query，**一律使用中文**（包含人名 / 作品名等固有英文专有名词的情况除外，例如 `久石让 Joe Hisaishi`）。百度后端对中文 query 命中率显著优于英文，Tavily 中英文均可。一律中文能保证在两个后端轮询时结果稳定。 |
| 2 | **采纳前必须读缩略图墙批量判定** | `web_search.py` 搜索后自动下载到 `workspace/downloads/` 暂存区并自动生成增量缩略图墙；**禁止**直接以哈希名（如 `img_5f3a8c1b.jpg`）在设计规范或 SVG 中引用。你必须：<br>  • **适合使用** → 用 `python scripts/web_search.py --adopt <downloads/xxx> <images/描述名>` 一次性完成移动+重命名，晋升到 `workspace/images/`；<br>  • **不合适** → 直接不管，交给 `--purge-downloads` 或自然淘汰；<br>  • **筛选后仍没有合适图片** → 换关键字继续搜索。 |

> 以上两条为**阻断性**约束：未完成图片审阅与重命名前，不得进入后续设计 / SVG 生成阶段。
>
> `web_search.py` 的完整 CLI、状态文件、配额、返回 JSON schema 见 [`references/web-search.md`](../references/web-search.md)。

## 步骤 1：主题确认

⛔ **阻断性步骤**：开始研究前，需与用户确认以下内容。如用户初始消息已清晰涵盖这些要点，则跳过确认直接进行。

| 项目 | 说明 | 示例 |
|------|------|------|
| **主题** | 核心主题 | 久石让 |
| **范围/重点** | 涵盖哪些方面 | 传记、主要作品、与宫崎骏的合作、获奖 |
| **深度** | 概览还是深入 | 通用知识级别 |
| **语言** | 输出语言 | 中文 |

**如用户请求已足够清晰**（如"做一个关于久石让的PPT"），推断合理默认值直接进行 — 不要过度询问。

## 步骤 2：内容研究

### 2.1 信息收集

**主工具** — `web_search.py`：

```bash
# 全局概览搜索
python scripts/web_search.py "<主题> 概述" -n 8 --json

# 定向补充搜索
python scripts/web_search.py "<子主题> 数据 统计" -n 5 --json
```

> 工具参数、配额、轮询机制见 [`references/web-search.md`](../references/web-search.md)。

**深度抓取** — 从搜索结果中识别出高信号 URL 后，使用 IDE 的 `WebFetch` 或 `scripts/source_to_md/web_to_md.py <URL>` 抓取完整页面内容。

| 来源类型 | 优先级 | 备注 |
|----------|--------|------|
| 官方网站 / 机构页面 | 高 | 第一方信息、可公开引用 |
| 国内百科（百度百科 / 搜狗百科 / 维基百科中文镜像） | 高 | 中文语境下的概览、时间线、关键事实 |
| 新闻 / 媒体文章 | 中 | 近期事件、获奖、公众反响 |
| 专业数据库 / 行业报告 | 低 | 需要时再取，注意可引用性 |

> Wikipedia / Wikimedia 在中国大陆通常无法直接访问，不作为默认来源；如调研环境允许再用。

**研究策略**：
1. 先运行一次宽范围的 `web_search.py` 了解主题全貌；
2. 对搜索结果里命中的权威页面用 `WebFetch` 抓 2–3 个获取详细内容；
3. 按需对特定子主题再次 `web_search.py` 定向补充。

### 2.2 内容组织

将收集的信息整理为结构化Markdown文档，骨架如下：

```markdown
# <主题名称>

> 主题的一句话简要描述。

## 概述
[2-3段摘要]

## 背景/历史
[时间线、起源、关键里程碑]

## 核心方面
### 方面1：[...]
### 方面2：[...]
### 方面3：[...]

## 成就/影响
[获奖、认可、影响力]

## 关键数据
| 项目 | 数值 |
|------|------|
| ... | ... |

## 来源
- [来源1标题](URL)
- [来源2标题](URL)
```

> 上述骨架为**参考模板** — 根据主题调整章节结构。人物传记与技术概览或公司简介的结构不同。

**内容准则**：
- 包含具体事实、日期和名称 — 避免模糊概括
- 保留找到的关键引述
- 注明数据来源以保证可验证性
- 追求PPT适用的内容密度：足够填充10-15页幻灯片，但非穷尽式研究论文

### 2.3 保存文档

将Markdown文档保存到 `workspace/sources/` 目录：

```
workspace/sources/<topic_name>.md
```

## 步骤 3：图片收集

### 3.1 图片来源策略

**主工具** — `web_search.py`：搜索后自动下载到 `workspace/downloads/` 暂存区，并在当下载批次累积至 ≥ 5 张时自动生成增量缩略图墙。LLM 无需手动拼图、无需手动 `curl`。工具细节见 [`references/web-search.md`](../references/web-search.md)。

补充策略——按以下优先级搜索**公开可用的免费图片**：

| 来源 | 查找方式 | 许可说明 |
|------|----------|----------|
| **官方网站 / 机构页面** | `WebFetch` 官方或机构页面 → 查找画廊或新闻区 | 通常可免费用于编辑/教育用途 |
| **政府 / 机构发布** | `web_search.py` 搜官方新闻资料包、公开画廊 | 通常为公共领域 |
| **新闻图库 / 媒体报道** | 从搜索结果取高信号 URL 后 `WebFetch` | 注意版权声明；仅用于不作商业用途的展示 |

**避免**：带水印的图库网站、有版权的商业图片、无明确许可的社交媒体上传。

> Wikipedia / Wikimedia Commons / Flickr Creative Commons 在大陆直接访问不稳定，不作为默认优先渠道；如调研环境允许再用。

### 3.2 图片选择标准

| 标准 | 指南 |
|------|------|
| **数量** | 6-12张（足以支撑10-15页PPT） |
| **多样性** | 人物像、场景、Logo、活动照片等混合 |
| **分辨率** | 优先宽度1000px+；避免缩略图 |
| **相关性** | 每张图片应有明确用途（封面、插图、背景） |
| **宽高比混合** | 适用时同时包含横版（背景用）和竖版（人物用） |

### 3.3 下载与审阅流程

#### 3.3.1 方式一：LLM自动审查（**推荐，简化流程**）

使用 `--review-images` 模式，LLM 自动审查图片并保存到 `workspace/images/`：

```bash
python scripts/web_search.py "<主题>" --review-images
```

**流程**：
1. 搜索并下载图片到内存（自动追加"图片 素材 插图 配图"关键词以过滤书籍等非图片内容）
2. LLM 逐个审查图片的相关性、质量、PPT适用性
3. 通过审查的图片自动保存到 `workspace/images/`，文件名为 LLM 生成的描述性英文名称
4. 返回审查摘要（发现/下载/通过/拒绝数量及文件列表）

**可选参数**：
- `--ppt-style`：指定PPT风格（如"高端咨询风"、"科技风"），LLM会根据风格调整审核标准
- `--ppt-audience`：指定目标受众（如"6-12岁儿童"、"企业高管"），LLM会根据受众调整审核标准
- 如果未指定，LLM会自动从 `workspace/spec_lock.json` 读取项目信息，或使用通用标准

**优势**：
- 跳过缩略图墙生成和手动采纳步骤
- LLM 直接生成描述性文件名
- 支持PPT上下文感知，审核更精准

详细说明见 [`references/web-search.md`](../references/web-search.md)。

#### 3.3.2 方式二：缩略图墙批量判定（**传统路径**）

1. 确认工作区目录已识别（`PPT_PROJECT_DIR` 或 `--project-dir`），关键字用中文（顶部约束 #1）；
2. 运行若干次 `web_search.py` 搜索，同一 URL / 同会话已采纳 URL 不会重复下载；
3. 读返回 JSON 的 `montage_batch.montages` 字段拿到本次新增的缩略图墙；若 `downloads_quota_error` 非空则按其提示处置（详细字段见 [`references/web-search.md`](../references/web-search.md)）；
4. **⛔ 读缩略图墙 + 采纳（阻断性）**：
   - 逐张读 `montage_batch_NN_*.jpg`，凭缩略图 + 文件名标签批量判定；对识别存疑的少数图，才需再开文件精查；
   - **适合使用** → `python scripts/web_search.py --adopt <downloads/xxx> <images/描述名>` 一行命令完成移动+重命名（命名规则见 3.3.4）；
   - **不合适** → 不管即可，下次迭代如果触配额再运行 `--purge-downloads` 清理；
   - **无任何图片合适** → 换一组中文关键字重新搜索，直至满足需求。

   > 完成本步前禁止进入步骤 4 或后续设计 / SVG 生成；未采纳的哈希名文件不得出现在 `spec_lock.json` 或 SVG 引用中（`finalize_svg.py` 会在后处理阶段硬门禁）。

#### 3.3.3 手动补充下载（兜底）

仅在自动下载失败、或从 `web_search.py` 以外的来源（官网 / 机构页面 / 国内百科等）取图时才需要。直接将文件以描述性名称放到 `workspace/images/` 即可，无需经过 downloads/ 暂存：

```powershell
curl -L -o "workspace/images/descriptive_name.jpg" "<image_url>"
```

#### 3.3.4 文件命名规则（采纳后必然执行，防止图片混用）

- **名称必须与图片实际内容一致**：凭缩略图墙的视觉内容归纳主体（人物 / 场景 / 对象 / 事件）与上下文（年份 / 地点 / 主题），组合为文件名；
- 示例：`joe_hisaishi_concert_tokyo_2023.jpg`、`spirited_away_poster.jpg`、`thyroid_ultrasound_front_view.png`；
- 反例（绝不得采纳，finalize 阶段会硬门禁）：`image1.jpg`、`img_5f3a8c1b.jpg`、`tmp_1.png`、`download.jpg`；
- 全小写，空格用下划线；同主体多版本用数字后缀区分（`joe_hisaishi_portrait_01.jpg`、`joe_hisaishi_portrait_02.jpg`）；
- 重命名后的文件名必须足以让后续设计阶段仅凭文件名就能判断其用途，不需再打开图片确认。

### 3.5 全分辨率图片备注

- 通过 `web_search.py` 自动下载的图片已默认取原始 URL，无须再做缩略图→全图的 URL 改写；
- 仅当手动从网页拷贝图片地址时，才需留意去掉 URL 中的 `thumb/`、`small`、`w=xxx` 等缩小后缀，避免拿到缩略版。

## 步骤 4：输出摘要

完成步骤2和3后，输出简要摘要：

```markdown
## 主题研究完成

**主题**: <topic_name>
**文档**: `workspace/sources/<topic_name>.md` — [X个章节，约Y字]
**图片**: `workspace/images/` — [N张图片]

| 文件名 | 来源 | 描述 |
|--------|------|------|
| ... | ... | ... |
```

> 此后，用户或主流水线可将这些材料作为PPT生成的输入（直接读取 `workspace/sources/` 中的 Markdown 文档）。
